# v0.9 Public Launch Runbook

This runbook is the shortest path from private repo to a credible public `v0.9.0` release.

Use this with:

- `docs/production-readiness-checklist.md`
- `docs/ci-branch-protection.md`
- `docs/post-public-finalization-checklist.md`

## 1) Local preflight before making repo public

From repo root:

```bash
make check
```

Optional (recommended) release-gate checks:

```bash
PYTHONPATH=. ./.venv/bin/python scripts/verticals/journalism/generate_sample.py --output /tmp/release_gate_sample.chronicle
PYTHONPATH=. ./.venv/bin/python scripts/run_conformance.py /tmp/release_gate_sample.chronicle
```

Done condition:

- Local checks are green and there are no known trust-surface regressions.

## 2) Make the GitHub repo public

GitHub UI:

1. `Settings` -> `General` -> `Danger Zone` -> `Change repository visibility`.
2. Select `Public` and confirm.

Done condition:

- Repository visibility shows `Public`.

## 3) Enable GitHub Actions

GitHub UI:

1. `Settings` -> `Actions` -> `General`.
2. Set workflow permissions to allow workflows in this repo to run.

Done condition:

- `.github/workflows/ci.yml` runs on push and pull request.

## 4) Configure branch protection on `main`

GitHub UI:

1. `Settings` -> `Branches` -> `Add branch protection rule`.
2. Branch pattern: `main`.
3. Enable:
   - Require a pull request before merging
   - Require status checks to pass before merging
   - Require branches to be up to date before merging
   - Include administrators
4. Disable:
   - Allow force pushes
   - Allow deletions
5. Required checks:
   - `lint-and-test (3.11)`
   - `lint-and-test (3.12)`
   - `frontend-checks`
   - `postgres-event-store-smoke`

Done condition:

- Rule is active and enforces the required checks above.

## 5) Validate CI and protection on a real PR

1. Open a small PR to `main`.
2. Confirm required jobs complete and merge is blocked if one fails.
3. Confirm direct pushes to `main` are blocked (if configured).

Done condition:

- Protection behavior is confirmed from PR UX, not only settings.

## 6) Produce machine-readable rollout evidence

Set env vars:

```bash
export GITHUB_REPOSITORY=owner/repo
export GITHUB_TOKEN=... # token with repo admin/read permissions
export CHRONICLE_PROTECTED_BRANCH=main
```

Run:

```bash
make branch-protection-rollout-check
```

Done condition:

- `reports/branch_protection_rollout_report.json` exists.
- Report has `"status": "passed"`.

## 7) Trigger manual gates and capture links

GitHub UI:

1. `Actions` -> run `Release (Manual)` for `v0.9.0` (validation mode first).
2. `Actions` -> run `Supply Chain (Manual)`.

Done condition:

- Manual gates are green.
- Links/artifacts are saved in release notes or evidence pack.

## 8) Finalize `v0.9.0` release

1. Create annotated tag and push:

```bash
git tag -a v0.9.0 -m "Chronicle v0.9.0"
git push origin v0.9.0
```

2. Publish GitHub release notes.
3. Update remaining checkboxes in `docs/to_do.md`.

Done condition:

- Public tag/release is live and all release blockers are closed.
